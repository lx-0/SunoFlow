## What this does

`pushJamPrompt(shareToken, {promptText, guestName?, guestKey})` mirrors the
canonical programmatic generation path (mcp/tools/generate_song.ts) but on
behalf of the HOST:

1. Session by token; must be status "open" (else 409 "session closed").
2. Prompt: stripHtml, 1..500 chars (Suno non-custom description cap).
3. Per-guest cap: max 2 entries with status "pending" per guestKey per
   session (slot frees when a song completes or fails) → RATE_LIMITED.
4. **Budget reservation is atomic**: `updateMany({where: {id, status:
   "open", budgetUsed: {lt: budgetTotal}}, data: {budgetUsed: {increment:
   1}}})` BEFORE the Suno call — two guests racing the last slot cannot
   overshoot. Suno/DB failure decrements back (compensation).
5. Host key via `resolveUserApiKeyWithMode(hostUserId)`; internal credits
   checked/deducted for non-personal-key hosts (checkCredits/deductCredits,
   MCP-tool pattern). Keyless demo mode mirrors the MCP tool (instant mock
   "ready" song) so the keyless E2E path works.
6. `generateSong(prompt, {}, hostKey)` → Song row (pending, owner = host,
   sunoJobId = taskId) + JamSessionEntry (pending, songId, promptText,
   guestName, guestKey) in one transaction.

Route `POST /api/jam/[token]/prompts` (anonRoute, 60/min IP backstop, zod
body promptText 1..500 / guestName ≤40 / guestKey 8..64) → 201 entry card.
