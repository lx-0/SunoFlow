# SunoFlow Mobile — UX/UI Audit (2026-07-17)

> Snapshot of a comprehensive UX/UI review of the Expo mobile app, run by six
> parallel review agents (navigation architecture, chrome/wayfinding, player UX,
> states/accessibility, design-system consistency, plus an exhaustive call-site
> inventory: 93 nav call sites, 57 screens) and hardened by three adversarial
> verifiers reading the vendored expo-router 56 source. All findings below were
> captured against the code BEFORE the navigation rework of the same day —
> navigation/chrome findings are therefore largely addressed (see "Fixed"), the
> rest is a prioritized backlog.

## Fixed in the navigation rework (same day)

- Real Tabs navigator with one stack per tab (shared 5-way route group with
  per-segment anchors) replaces the flat root Stack + dismissAll()+navigate()
  pseudo-tabs. Instant tab switches, per-tab back stacks + scroll positions,
  predictable Back, iOS swipe-back everywhere.
- Tab presses dispatch NAVIGATE-by-name / POP_TO_TOP at the navigator level
  (stock tab-bar actions); re-tap pops to the tab anchor. Active-tab highlight
  comes from navigator state (was: dead on ~90% of screens).
- Profile is a real tab (hamburger, no back chevron).
- Player modal is headerless (double chrome removed), swipe-down dismiss;
  queue/lyrics/add-to-playlist are sheets over it with explicit close buttons;
  content exits go through closePlayerThen() with group-qualified hrefs
  (dismiss + push land on the ACTIVE tab).
- Chrome (tab bar + mini-player) lives inside the Tabs layout — no more
  pathname-gated GlobalChrome popping in/out; tab bar renders in-flow, so
  content never scrolls under it.
- Sidebar edge-swipe restricted to tab roots (was fighting iOS swipe-back).
- Bottom clearance: settings scrollable, search/playlist-detail/rss-feeds/
  profile got MINIPLAYER_CLEARANCE padding.
- Async poll redirects (generate/mashup/upload) gate on focus and defer until
  the screen is focused again.

**Runtime status: UNTESTED on device.** `tsc` clean, `expo lint` 0 errors, but
navigation is runtime behavior — walk the checklist in `NAVIGATION.md` on the
next dev build.

   feed, generations, radio, notifications, inspire, recommendations); list is

## Top remaining backlog (not addressed, by impact)

1. **Playback store emits unconditionally every 700ms** — every subscriber
   re-renders, progress UI steps (player, medium; known deferred issue).
2. **Accessibility near-zero outside the player**: unlabeled icon buttons,

3. **Keyboard handling**: KAVs lack keyboardVerticalOffset under native

## States, feedback + accessibility

The mobile app has a genuinely strong states-and-feedback backbone — EmptyState is used on 37 screens, most mutations show Alert/inline errors, optimistic updates revert, and player.tsx is a model of a11y labeling — but the quality is uneven exactly where users live most. The highest-impact defects are structural: the bottom-clearance constant (96) predates the global tab-bar+miniplayer overlay so nearly every scroll surface hides its last rows behind chrome (and Settings isn't scrollable at all, burying Sign out); there is no pull-to-refresh anywhere despite UI copy telling users to pull; and the primary Library tab plus Mashup have the worst states in the app (bare dim-text error≈empty with no retry, and an error that renders as an infinite spinner). Accessibility outside the player is near-zero — 4 files with roles, 0 uses of accessibilityState, unlabeled icon-only buttons throughout, a drawer that isn't modal to VoiceOver — and sub-44pt touch targets cluster in the chip rows and Inspire's ~20pt primary card action. Keyboard handling is systematically off: every KAV lacks keyboardVerticalOffset under the native header, and profile/upload/mashup/login have no keyboard avoidance at all.

### [HIGH] MINIPLAYER_CLEARANCE (96) no longer clears the global chrome stack — list/form bottoms are obscured

`apps/mobile/src/components/MiniPlayer.tsx:21`

src/components/MiniPlayer.tsx:21 defines MINIPLAYER_CLEARANCE=96 with the comment 'bar height (56) + gap + margin'. But since the global-chrome rework (app/_layout.tsx:53-62), every non-player screen has BottomTabBar (49pt content + insets.bottom, BottomTabBar.tsx:30) overlaid PLUS the MiniPlayer floating above it at bottom = insets.bottom + 49 + 10 (MiniPlayer.tsx:35). With a song playing on a notched iPhone the top of the MiniPlayer sits ~149pt above the physical bottom, yet 43 screens pad scroll content with only 96 (e.g. app/(tabs)/index.tsx:229, app/feedback.tsx:122, app/generate.tsx:604). The last list row / the 'Save as preset' button ends up under the chrome even when fully scrolled. Values also drift per screen: song/[id].tsx:535 uses 140, mashup.tsx:178 hardcodes 96, and app/search.tsx:131 SectionList has NO bottom padding at all — its last result is always hidden while something plays. This is the known past 'bottom-bar clearance' issue re-introduced by the tab-bar overlay.

**Recommendation:** Compute clearance at runtime: expose a useBottomClearance() hook = insets.bottom + TAB_BAR_HEIGHT + BOTTOM_GAP + 56 + margin (and drop the tab-bar/miniplayer parts on screens where chrome is hidden or idle). Replace all hardcoded 96/140 paddings and add the missing contentContainerStyle to search.tsx's SectionList.

### [HIGH] Settings screen is not scrollable and its bottom actions sit behind the tab bar

`apps/mobile/app/(tabs)/settings.tsx:63`

app/(tabs)/settings.tsx:63 renders ~15 rows in a plain View (no ScrollView), and the Sign out button uses marginTop:'auto' (line 161) so it is pinned to the container bottom — which extends under the absolutely-positioned global BottomTabBar (49pt + safe area) and, when audio plays, the MiniPlayer. Sign out / Delete account become partially or fully covered; on smaller devices (SE/13 mini) the section list overflows with no way to scroll. Every other content screen uses ScrollView/FlatList with (attempted) clearance.

**Recommendation:** Wrap the content in a ScrollView with bottom clearance padding (see finding 1) and remove marginTop:'auto' in favor of normal flow spacing.

### [HIGH] Mashup library-load error is never displayed — user sees an infinite spinner

`apps/mobile/app/mashup.tsx:157`

app/mashup.tsx:46 sets setError(...) when fetchLibrary fails, but the 'form' phase render (lines 133-170) never shows `error` — it is only rendered in the 'failed' phase (line 125), which is reached solely via submit. ListEmptyComponent (lines 157-161) shows an ActivityIndicator whenever songs===null, so a failed load spins forever with zero feedback and no retry path. This violates the app's own 'every path ends in visible feedback' rule stated in login.tsx.

**Recommendation:** In the form phase, branch on error before the songs===null spinner: render EmptyState tone="error" with a Retry CTA that re-runs the load.

### [HIGH] Library (the primary tab) has the weakest states in the app: bare dim-text error/empty, no retry, no focus reload

`apps/mobile/app/(tabs)/index.tsx:214`

app/(tabs)/index.tsx:214-219: both the error state and the empty state are a single dim Text inside Centered — visually identical to each other (error looks like empty and vice versa), with no retry CTA. Favorites/History/Playlists/Search all use EmptyState with icon + tone="error" + Retry. The empty copy 'No songs yet. Generate some on the web.' (line 219) is also stale — the app has its own /generate screen. Additionally, unlike favorites.tsx:36 and history.tsx:35, Library never reloads on focus, so a song generated in-app doesn't appear until a manual search/sort change.

**Recommendation:** Use EmptyState for both branches (error: tone="error" + Retry onCta=load; empty: CTA 'Generate a song' → /generate), and add useFocusEffect stale-while-revalidate like the sibling tabs.

### [HIGH] No pull-to-refresh anywhere in the app (grep: zero RefreshControl), and profile copy references a refresh gesture that doesn't exist

`apps/mobile/app/profile.tsx:292`

grep across app/ and src/ finds no RefreshControl or onRefresh at all. Long-lived lists (Library index.tsx:221, Discover discover.tsx:151, Notifications notifications.tsx:94, Feed, Generations) rely on focus-refetch or nothing; a stale or failed list can only be recovered by leaving the screen. Worse, profile.tsx:292's secondary-error notice literally says "Couldn't load some stats. Pull back later to retry." — telling the user to pull on a ScrollView that has no RefreshControl.

**Recommendation:** Add RefreshControl (tint colors.accent) to the main FlatList/ScrollView screens — at minimum Library, Discover, Notifications, Playlists, Favorites, History, Profile — wired to the existing load() callbacks.

### [MEDIUM] Tapping a song that fails to play gives zero visible feedback (silent catch in async tap handlers, repeated 6+ times)

`apps/mobile/app/(tabs)/index.tsx:177`

The play-on-tap handlers end in console.error only: app/(tabs)/index.tsx:177 (play failed), favorites.tsx:72, history.tsx:69, search.tsx:149, discover.tsx:175, song/[id].tsx:327 and 514. If playQueue rejects (network, expired stream URL), nothing happens on screen — the user taps repeatedly with no error. Same pattern for pagination: index.tsx:122-125 silently sets hasMoreRef=false so infinite scroll just stops, and discover.tsx:97-99 swallows load-more errors.

**Recommendation:** Route all play failures through one helper that shows visible feedback (Alert or a toast: "Couldn't play — check your connection"), and show a tappable 'Couldn't load more — retry' footer instead of silently ending pagination.

### [MEDIUM] All KeyboardAvoidingViews lack keyboardVerticalOffset under the native Stack header; several form screens have no keyboard avoidance at all

`apps/mobile/app/profile.tsx:261`

grep finds zero keyboardVerticalOffset in the codebase. Every iOS KAV uses behavior="padding" while sitting under a native-stack header (generate.tsx:355, feedback.tsx:66, change-password.tsx:49, comments/[id], lyrics-edit/[id], replace-section/[id], song-tags/[id]) — so the padding is short by the header height (~100pt) and bottom fields/buttons (generate's lyrics input at line 524, feedback's Send at line 111) are partially covered by the keyboard. Worse, three input-heavy screens have no KAV at all: profile.tsx:261 (7 TextInputs, the Preferences card near the bottom is fully covered while editing), upload.tsx:120 (URL + title inputs), and mashup.tsx:145-146 (title/style inputs in a FlatList header). login.tsx:44 also has none — on small devices the Sign in button can hide behind the keyboard.

**Recommendation:** Add keyboardVerticalOffset={headerHeight} (useHeaderHeight() from @react-navigation/elements) to every KAV under a header, and wrap profile/upload/mashup/login in the same KAV+ScrollView pattern used by feedback.tsx.

### [MEDIUM] Accessibility coverage is near-zero outside player.tsx: unlabeled icon-only buttons, no roles, no accessibilityState anywhere

`apps/mobile/src/components/Sidebar.tsx:42`

Grep: accessibilityRole appears in only 4 files, accessibilityLabel in 8, accessibilityState in 0. Unlabeled icon-only controls include: playlists.tsx:127-131 (play/shuffle per-row buttons), index.tsx:209 (grid/list toggle), inspire.tsx:86 (headerRight refresh), song/[id].tsx:374 (favorite heart) and 378 (share), Sidebar.tsx:42 (hamburger SidebarToggle — the primary nav affordance), Sidebar rows (line 154, no role), ReactionPicker.tsx:49 trigger + 35-47 emoji buttons, SongRow.tsx:26 (no role; isFavorite state not conveyed), MiniPlayer.tsx:34 outer bar (no 'Open player' label), feedback.tsx:99 rating stars, RatingStars.tsx:44 stars. Selected/active state is never conveyed: BottomTabBar.tsx:35-41 has role+label but no accessibilityState={{selected}}, same for generate.tsx persona/count chips, discover mood chips, upload/feedback segments, thumbs buttons. Waveform.tsx:109's seek Pressable is invisible to VoiceOver as a slider (no role='adjustable'/value).

**Recommendation:** Sweep: give every icon-only Pressable accessibilityRole="button" + accessibilityLabel; add accessibilityState={{selected/checked}} to tabs, chips, segments and thumbs; make Waveform an adjustable element with accessibilityValue and increment/decrement seek actions.

### [MEDIUM] Sidebar drawer is not modal to assistive tech and its backdrop is unlabeled

`apps/mobile/src/components/Sidebar.tsx:141`

src/components/Sidebar.tsx:141-163: the open drawer is a plain absolute-fill View — no accessibilityViewIsModal on the panel, no importantForAccessibility="no-hide-descendants" on the content behind it, so VoiceOver users can swipe focus into the obscured screen behind the open drawer. The dismiss backdrop (line 144) is a bare Pressable with no accessibilityLabel/role ("Close menu"). The edge-swipe catcher (line 130) is likewise a silent 22pt strip with no accessible alternative announced.

**Recommendation:** Set accessibilityViewIsModal on the panel Animated.View, label the backdrop Pressable ("Close menu", role button), and announce open/close via AccessibilityInfo.announceForAccessibility.

### [MEDIUM] Sub-44pt touch targets across chip rows, mini-buttons and card actions; worst is Inspire's ~20pt 'Generate from this'

`apps/mobile/app/inspire.tsx:184`

Measured from styles: inspire.tsx:184 genBtn has no vertical padding — the card's ONLY action ('Generate from this', line 151) is ~20pt tall with no hitSlop; inspire.tsx:109 'Manage RSS feeds' is a bare 14pt text link. generate.tsx:623 miniBtn (Boost/Auto-fill/Generate-lyrics) ≈33pt; generate.tsx:610 preset/persona/suggestion chips ≈32pt; countChip 40pt (line 616). index.tsx:277 sort/filter buttons ≈28pt and viewBtn 36pt (line 281), all without hitSlop. discover.tsx:217 mood chips ≈30pt. EmptyState.tsx:46 CTA ≈42pt. RatingStars.tsx:48 stars are 36pt wide cells (28pt icon + 4pt padding) with only hitSlop 6.

**Recommendation:** Establish a minHeight:44 rule for tappables (or add hitSlop to reach 44×44); prioritize Inspire's genBtn/manage link, the generate mini-buttons, and the Library sort/filter/view controls.

### [MEDIUM] song/[id] resets to a full-screen spinner on every refocus, and its sub-loads fail silently into fake-empty states

`apps/mobile/app/song/[id].tsx:50`

app/song/[id].tsx:50-56 (useFocusEffect) nulls song/lyrics/related/tags/thumbs on every focus — returning from a pushed child (Edit tags, Lyrics edit, Versions…) flashes the whole screen back to a centered spinner and re-scrolls to top: heavy layout shift on a hub screen users bounce in and out of. Lines 69-72 swallow lyrics/related/tags/feedback errors with .catch(() => {}) — a failed lyrics fetch renders as 'song has no lyrics' (section simply missing), indistinguishable from real absence. Same silent pattern in player.tsx:51-52 (favorite/reactions). The error state at song/[id].tsx:290-296 is bare dim text with no retry, inconsistent with EmptyState elsewhere.

**Recommendation:** Keep previous data on refocus (stale-while-revalidate like favorites.tsx) instead of nulling; surface sub-load failures as inline 'Couldn't load lyrics — retry' rows; use EmptyState + Retry for the main error branch.

### [MEDIUM] Error EmptyStates mostly lack a Retry CTA, and tone="error" differs from empty only by icon tint

`apps/mobile/src/components/EmptyState.tsx:29`

EmptyState.tsx:29: tone="error" only switches the icon color to danger — title/subtitle/layout are identical to the empty look, so at a glance an error reads as 'you have no content' (the exact web-app bug recorded in the UX audit). Several error usages pass no ctaLabel/onCta, leaving dead ends with no recovery: search.tsx:123, (tabs)/playlists.tsx:89, discover.tsx:139, notifications.tsx:82, lyrics.tsx:71, add-to-playlist.tsx:68. Favorites/History do it right (Retry CTA, favorites.tsx:41-48).

**Recommendation:** Make tone="error" visually distinct (danger-tinted title or badge) and require/auto-provide a Retry CTA for error tone; thread the existing load() callbacks into the six CTA-less error sites.

### [LOW] Library action sheets hardcode userInterfaceStyle:"dark", breaking light mode; other sheets omit it inconsistently

`apps/mobile/app/(tabs)/index.tsx:132`

app/(tabs)/index.tsx:132 and 152 pass userInterfaceStyle:"dark" to ActionSheetIOS while the app ships a full light mode + system mode (settings.tsx MODES, ThemeContext). In light mode these two sheets render dark while every other ActionSheetIOS call in the app (generate.tsx:126, song/[id].tsx:192/267/284, profile.tsx:193) follows the system appearance — so sheets are inconsistent among themselves and with the chosen theme.

**Recommendation:** Drop the hardcoded value or derive it from useTheme().scheme in one shared helper for all ActionSheetIOS call sites.

### [LOW] textFaint fails WCAG AA contrast for the small text it styles (tab labels, sidebar section headers, grid subtitles)

`apps/mobile/src/theme/theme.ts:44`

src/theme/theme.ts:44 dark textFaint #686164 on surface #151012 ≈ 3.5:1, and theme.ts:62 light textFaint #8a8487 on #faf8f9 ≈ 3.4:1 — both below the 4.5:1 AA threshold for small text. It is used on 11-13pt text: inactive BottomTabBar labels (BottomTabBar.tsx:33 tint + :43 label, 11pt), Sidebar section titles (Sidebar.tsx:184, 11pt), library grid subtitles, placeholders and EmptyState icons. Inactive tab labels are primary navigation and should stay readable.

**Recommendation:** Lighten textFaint toward ~#7d767a (dark) / darken toward ~#6f696c (light) to clear 4.5:1, or use textDim for functional small text and reserve textFaint for decorative elements.

### [LOW] Add-to-playlist header shows the currently-playing title instead of the target song; passed title param is ignored

`apps/mobile/app/add-to-playlist.tsx:79`

app/add-to-playlist.tsx:22 resolves songId from params (e.g. deep-linked from song detail with `songId` + `title`, see song/[id].tsx:334), but the list header at line 79 renders `Add "{current?.title}" to…` — the *playing* song's title. When adding a non-playing song the header names the wrong song (or is blank), so the confirmation context misleads. The `title` param declared at line 21 is never used. Otherwise this screen has the app's best per-row feedback ('Failed, tap to retry', lines 88-90).

**Recommendation:** Prefer paramStr(params.title) ?? current?.title in the header so the named song always matches the one being added.

### [LOW] Decorative data loads fail silently with inconsistent recovery: settings credits stuck on '…', generate's picker rows just vanish

`apps/mobile/app/generate.tsx:104`

app/(tabs)/settings.tsx:27 — fetchCredits failure is console.error only, leaving the Credits row showing '…' indefinitely (line 101) with no retry. app/generate.tsx:104-109 — presets, prompt templates, personas, suggestions and trending all swallow errors and silently hide their sections, while style templates alone got an explicit error status + 'Tap to retry' affordance (lines 451-454). A user whose personas fail to load has no hint the section exists. Same class: root auth check (_layout.tsx:67-73) swallows getApiKey failure.

**Recommendation:** Reuse the stStatus pattern (loading/ready/error + tap-to-retry) for the other generate pickers, and render 'Couldn't load — retry' in the settings Credits row on failure.

### [LOW] Reduced motion honored in only 3 places; modal slide animations and the 2s reaction fade ignore it

`apps/mobile/src/components/PromptSheet.tsx:72`

useReducedMotion is consumed only by Sidebar.tsx:123 (slide → 0ms, good), lyrics.tsx:63 (auto-scroll, good) and Waveform.tsx:29 (drops popup travel but keeps the 2000ms Animated fade at line 26 running regardless). Not consulted by: PromptSheet.tsx:72 Modal animationType="slide", player.tsx:204 menu Modal slide, and the player screen's modal presentation. iOS reduce-motion does not automatically suppress RN Modal slide animations.

**Recommendation:** Pass animationType={reduceMotion ? "fade" : "slide"} to both Modals and shorten/skip the reaction popup timing under reduce-motion for full PRODUCT.md compliance.


## Navigation architecture

The mobile app's navigation model is a flat root Stack wearing a tab bar as a costume: the five "tabs" are Stack screens switched via dismissAll()+navigate(), so every tab change is one or two animated push/pop transitions routed through Library, with full unmount/remount of the target, no per-tab back stacks, and — because dismissAll only collapses the closest stack — Back destinations that differ depending on the path taken (swipe-back from Favorites can land on Playlists or Library). The unnatural feel the user reports is over-determined: state-wiping useFocusEffect reloads make every return (including dismissing the player) a spinner at scroll-top; the Profile tab is a root-pushed screen with a Back chevron; the sidebar's left-edge-swipe PanResponder fights iOS swipe-back on the same edge; inner tab screens hide their back button behind a hamburger while the pop gesture stays live; and player-menu pushes build a second, chrome-less navigation context on top of the Now-Playing modal. NAVIGATION.md is internally coherent but codifies non-native semantics ("Back returns to the home tab") and is already drifting (song-detail Extend and notification section-targets use raw push), and its own on-device verification checklist is entirely unchecked. The durable fix is structural, not more call-site discipline: a real tab navigator with one stack per tab, player peeks scoped to the modal, and query-cached screen data — after which switchTo/goToSection and most of the special-casing can be deleted.

### [HIGH] Tabs are not tabs: (tabs) is a Stack, so tab switches are animated pushes/pops routed through Library

`apps/mobile/app/(tabs)/_layout.tsx:13`

app/(tabs)/_layout.tsx:13-26 declares the five 'primary' screens as a plain expo-router Stack, and BottomTabBar.tsx:38 switches between them via switchTo() -> goToSection() -> dismissAll()+navigate() (src/navigation.ts:42-51). Concretely: on Playlists, tapping Favorites first animates a popToTop back to Library (index), then animates a push of Favorites — two queued full-screen slide transitions with Library momentarily visible in between. Native iOS tab bars (Apple Music/Spotify, UITabBarController) switch instantly with no horizontal push animation, keep every tab's subtree mounted, and give each tab its own back stack. Here each non-home tab is unmounted by every dismissAll and re-pushed, so there is exactly one shared stack whose base is always Library. This single decision is the primary root cause of 'navigation feels unnatural': section changes read as page navigations, not tab switches. NAVIGATION.md:56-63 claims this is 'the Spotify / Apple Music pattern', but only the persistent chrome matches; the transition/state semantics do not. Note also NAVIGATION.md:122-133 — every runtime verification checkbox is still unchecked.

**Recommendation:** Replace the (tabs) Stack + custom overlay bar with a real tab navigator (expo-router Tabs / native BottomTabNavigator, or expo-router's NativeTabs) with one Stack per tab: instant switch, per-tab state preservation, per-tab back stacks. Keep the custom visual design via a custom tabBar renderer if the styled bar is wanted. Non-tab sections (Discover, Generate, ...) can stay as root-stack pushes over the tab navigator.

### [HIGH] Back is unpredictable: destination depends on path taken, and often is Library instead of where the user came from

`apps/mobile/src/navigation.ts:42`

Because dismissAll() only collapses the CLOSEST stack (src/navigation.ts:43-49), the inner (tabs) stack and the root stack are collapsed on different taps, producing inconsistent Back targets. Repro A: Library -> Playlists (inner stack [index,playlists]) -> Sidebar Discover (dismissAll pops inner to Library, pushes /discover on root) -> Back => Library, NOT Playlists where the user actually was. Repro B: Library -> Playlists -> Discover -> tab Favorites (dismissAll pops ROOT to (tabs), whose inner stack is still [index,playlists]; navigate pushes favorites onto it => [index,playlists,favorites]) -> iOS swipe-back from Favorites => Playlists. Same gesture from Favorites reached via Library->Favorites => Library. So swipe-back from the identical screen lands on different screens depending on history, directly violating iOS's 'back returns where you came from' expectation, and contradicting NAVIGATION.md:70-73's claim that the stack is 'at most [home base] -> [one section]'. In a real tab model this whole class disappears because sections are parallel, not stacked.

**Recommendation:** Adopt per-tab stacks (see finding 1). If the flat model is kept short-term, make switchTo collapse BOTH stacks (dismiss root to (tabs), then popToTop the inner stack) so Back is at least consistently 'Library', and document that as an explicit non-iOS trade-off.

### [HIGH] dismissAll+navigate remounts screens and the useFocusEffect-reload pattern wipes scroll/data on every return — including after closing the player

`apps/mobile/app/discover.tsx:69`

Every section switch unmounts the previous section (dismissAll), so scroll position, pagination, filter state and fetched data are destroyed; returning re-runs the full load with a spinner. Worse, most screens compound this by resetting state on FOCUS, not just mount: app/discover.tsx:69-73 runs load() in useFocusEffect which does setSongs(null) (line 53), so even a plain Back into Discover — or merely dismissing the Now-Playing modal opened from a Discover row (discover.tsx:170-177) — resets a 5-page-deep browsing session to page 1, scroll top, with a loading spinner. app/song/[id].tsx:49-77 does the same (setSong(null) + 5 refetches on every refocus), so returning from any pushed child (Edit tags, Lyrics edit, Related) rebuilds the whole detail screen. In Apple Music/Spotify, dismissing Now Playing or popping a detail returns you to a pixel-identical, still-scrolled list. This turns the core listen-browse loop into a state-loss loop and amplifies every other nav defect.

**Recommendation:** Keep tab/section subtrees mounted (real tab navigator) and switch data loading to a stale-while-revalidate cache (e.g. TanStack Query) keyed by route params: refetch on focus is fine, but never null-out existing data or reset scroll. Reserve hard resets for param changes only.

### [HIGH] Profile 'tab' is a root-level pushed screen with a Back chevron — a tab you can swipe-dismiss

`apps/mobile/src/components/BottomTabBar.tsx:20`

BottomTabBar.tsx:20 lists Profile as the fifth tab, but /profile is an auto-registered root Stack screen (app/profile.tsx; not in app/(tabs)/), inheriting the root screenOptions with headerBackButtonDisplayMode:'minimal' (app/_layout.tsx:83) and no headerLeft override. Result: tapping the Profile TAB pushes a screen that (a) slides in horizontally, (b) shows a Back chevron in the header pointing to Library, (c) can be swipe-back-dismissed — after which the tab highlight silently jumps to whatever (tabs) screen is beneath, and (d) lacks the hamburger SidebarToggle that the other four tab screens get from (tabs)/_layout.tsx:18, so header affordances differ between tabs. Also, because switchTo first collapses the inner stack, tapping Profile from History and then Back lands on Library, not History. No native iOS tab ever shows a back button to another tab. Additionally, profile.tsx:79-131 reloads six endpoints on every focus, so each visit to the 'tab' is a spinner.

**Recommendation:** Move profile into the tab navigator as a first-class tab root (app/(tabs)/profile.tsx) with the shared header treatment, or drop it from the tab bar and keep it sidebar-only. A screen must not be simultaneously a tab destination and a pushed stack child.

### [MEDIUM] Left-edge swipe is overloaded: sidebar-open PanResponder competes with iOS interactive swipe-back on the same edge

`apps/mobile/src/components/Sidebar.tsx:111`

## Design-system consistency

The mobile theme layer (src/theme/theme.ts) is a faithful, well-documented sRGB conversion of DESIGN.md's OKLCH palette, and color discipline is structurally good: only ~14 hardcoded hex literals exist outside the theme, and status pills in generations.tsx follow the status-as-verbs language. However, the two most identity-defining rules of the spec are broken app-wide: Geist Sans is loaded (and first paint gated on it) but never applied to a single Text element, so 100% of chrome renders in the iOS system font; and virtually every primary CTA (41 sites, including the Generate button) uses the deep accentStrong fill with near-black text at a failing 3.0:1 contrast instead of the spec's Electric Magenta pairing (4.84:1), while the one spec-mandated magenta surface, the waveform's played portion, renders white instead. Below those, the system shows token-gap-driven drift: no radius/spacing/surface-hover/info tokens exist, producing 20 ad-hoc radii, four chip recipes, inputs and cards split across two tonal planes, and accent overuse in the player menu and playlist rows. Most findings are mechanical to fix once shared primitives (themed Text, Button, Chip, radius tokens) exist; without them the per-screen makeStyles pattern will keep re-drifting.

## Player + Now-Playing UX

The player/now-playing layer is functionally rich (waveform seek, timed reactions, shuffle-versions, queue editing) but its presentation model is the main source of the "unnatural navigation" complaint: the Now-Playing sheet carries a doubled header (native "Now Playing" bar plus a custom chevron row) instead of full-bleed art with a grabber, and every peek pushed from its menu is silently re-presented by expo-router's modal-after-modal rule as another stacked sheet with no back or close button — the documented "Back returns to the player" pattern does not actually exist, and the same routes transition differently depending on origin. The root-overlay chrome (GlobalChrome pathname check) can never coexist with native modals, so the tab bar and mini-player pop out instantly on open and blink back only after an interactive dismissal completes, while MINIPLAYER_CLEARANCE under-measures the real occlusion by ~50px, hiding the last row of every list during playback. Smoothness is further capped by the 700ms unconditional store emit (all subscribers re-render, progress steps in visible jumps) and papercuts like tap-only seeking with poll snap-back, lyrics auto-scroll that fights the user, and a wrong-song title in Add-to-Playlist. Most of these converge on the already-planned restructure (real tabs navigator + a nested stack inside the player modal + selector-based playback subscriptions); the add-to-playlist title, openPlayer() consolidation, and clearance math are cheap standalone fixes. All findings are code-derived (expo-router 56.2.8 / react-native-screens 4.25.2 sources verified) but not device-verified.

### [HIGH] Player-menu 'peek' screens silently become stacked sheets with no back/close affordance

`apps/mobile/app/player.tsx:77`

Every peek pushed from the player "..." menu (app/player.tsx:77-86: /lyrics, /queue, /add-to-playlist, /comments/[id], /related/[id], /song/[id], /song-versions/[id]) uses router.push while the /player modal (app/_layout.tsx:88) is in the root stack. expo-router's vendored react-navigation fork (node_modules/.pnpm/expo-router@56.2.8_*/node_modules/expo-router/build/react-navigation/native-stack/utils/getModalRoutesKeys.js: `(acc.length && !presentation)`) classifies any route WITHOUT an explicit `presentation` that appears after a modal route as a modal itself, and SceneView then forces `presentation = 'modal'` (views/NativeStackView.native.js:59). react-native-screens 4.25.2 confirms the alternative would be worse: RNSScreenStack.mm:690-737 puts push-presentation screens into the UINavigationController UNDER the presented modals. Net effect: each peek opens as another pageSheet stacked on the Now-Playing sheet, its native header renders with NO back button (modal = root of its own presentation; headerBackButtonDisplayMode 'minimal' at app/_layout.tsx:83 is irrelevant), so the only way back to the player is the undiscoverable swipe-down gesture. NAVIGATION.md:113-115 claims "Back returns to the player (the now-playing-peek pattern)" — that Back affordance does not exist. Worse, the SAME routes (/song/[id], /add-to-playlist via song detail) present as regular card pushes with a back chevron when reached without the player open, so identical destinations get different transitions and different dismissal mechanics depending on origin — a direct source of the "navigation feels unnatural" complaint.

**Recommendation:** Give the player modal its own nested stack (app/player/ layout or the planned Tabs restructure) so peeks are true pushes inside the sheet with headers and back buttons; alternatively add an explicit headerLeft close/back button to every peek screen and declare their presentation explicitly instead of inheriting modal-after-modal.

### [HIGH] Now-Playing modal has a double header and no full-bleed layout, contra native music-app convention

`apps/mobile/app/_layout.tsx:88`

app/_layout.tsx:88 declares the player with `presentation: "modal", title: "Now Playing"` and never sets headerShown:false, so the sheet renders a native navigation bar titled "Now Playing" (with no close button — modals get no back item). Directly beneath it, app/player.tsx:107-111 renders a second custom header row containing only a ChevronDown that calls router.back(). The result is two stacked header bars eating ~90px of vertical space above the 300px cover art, with the visible nav bar offering no dismiss affordance and the actual dismiss control sitting in a redundant second row. Native convention (Apple Music/Spotify) is full-bleed artwork, a grabber, and swipe-to-dismiss; here `sheetGrabberVisible` defaults to false (vendored NativeStackView.native.js:60) so no grabber renders either. openPlayer() via router.navigate (src/navigation.ts:28-30) presents a standard pageSheet slide-up — acceptable — but the doubled chrome is what makes the modal read as a generic form sheet instead of a Now-Playing surface.

**Recommendation:** Set headerShown:false on the player screen and keep only the in-screen chevron (or a grabber via sheetGrabberVisible), letting the cover art go full-bleed; keep swipe-to-dismiss as the primary gesture.

### [MEDIUM] GlobalChrome pathname check makes tab bar + mini-player pop in/out instead of animating with the player modal

`apps/mobile/app/_layout.tsx:53`

GlobalChrome (app/_layout.tsx:53-62) unmounts MiniPlayer+BottomTabBar the instant `usePathname()` becomes "/player". **Recommendation:** Move the chrome inside the (tabs) navigator context (part of the planned Tabs restructure) so it stays put during modal transitions, or at minimum animate its opacity with the transition instead of unmounting on a pathname string match.

### [MEDIUM] MINIPLAYER_CLEARANCE (96) does not clear the tab bar + safe area, so the last list rows sit behind chrome while music plays

`apps/mobile/src/components/MiniPlayer.tsx:21`

MiniPlayer.tsx:35 positions the bar at `insets.bottom + tabBarHeight + 10` (34+49+10 = 93px on notched iPhones) with height 56, so the occluded band reaches 149px from the screen bottom (83px tab bar + 10px gap + 56px bar). **Recommendation:** Compute clearance as TAB_BAR_HEIGHT + insets.bottom + gap + 56 + margin (a hook, since insets are runtime), or stop overlaying and give screens real bottom space via the tabs navigator.

### [MEDIUM] Playback store emits a new snapshot unconditionally every 700ms, re-rendering every mounted subscriber and stepping all progress UI

The poll (src/playback/audio.ts:103-129) calls patch() each tick; patch (audio.ts:69-72) always creates a new snapshot object with no change detection, so useSyncExternalStore re-renders every subscriber every 700ms — even while PAUSED with nothing changing. ### [LOW] Player menu closes its RN Modal and navigates in the same synchronous call — iOS present/dismiss race

`apps/mobile/app/player.tsx:88`

runMenu (app/player.tsx:88-91) does setMenuOpen(false) then go() immediately.
